## ข้อสรุปการเชื่อมต่อและ Protocol ระหว่าง ROS2 กับ STM32 / Arduino

### 1. โครงสร้างปัจจุบันของฝั่ง STM32 (อ้างอิงจากโค้ดที่มีอยู่)
จากการตรวจสอบไฟล์ทั้งหมดในโปรเจกต์ (รวมถึง `main.cpp`, `main.txt`, `motor_control.cpp`, `engine_control.cpp` และไฟล์อื่นๆ ในโฟลเดอร์ `src/` และ `include/`) **ยังไม่มีการเขียนโค้ดสำหรับ "รับคำสั่ง" (รับข้อมูลเข้าผ่าน Serial) ไว้อยู่ใน STM32 เลยครับ** มีเพียงแค่การส่งข้อมูลออก (Transmit ข้อมูลจำพวก IMU, Encoder, Ultrasonic) เท่านั้น ดังนั้นเราจำเป็นต้องเขียนส่วนการรอรับข้อมูลผ่าน `Serial1.available()` และแยกแยะข้อความเพิ่มเข้าไปใน STM32 ครับ

ฟังก์ชันที่มีอยู่แล้วในฝั่ง STM32 พร้อมถูกเรียกใช้งานเมื่อได้รับคำสั่ง มีดังนี้:
- `driveMotors(int speedL, int speedR)`: ควบคุมสายพานซ้ายและขวา รับค่าช่วง `-100` ถึง `100`
- `setThrottle(int percent)`: ควบคุมคันเร่ง (Servo) รับค่าช่วง `0` ถึง `100`
- `startEngine()`, `powerOn()`, `powerOff()`: สั่งงานเครื่องยนต์/รีเลย์

---

### 2. ทางแก้และรูปแบบ Protocol ที่นำเสนอ (สำหรับเขียนเพิ่มใน STM32 และ ROS2)
เนื่องจากเราต้องเพิ่มการรับข้อมูลผ่าน `Serial1` บน STM32 เสนอให้ใช้ Protocol เป็น String ที่ง่ายต่อการแยกแยะ (เหมือนกับสิ่งที่ STM ส่งหา ROS) ดังนี้ครับ:

**รูปแบบคำสั่งส่งจาก ROS2 -> STM32:**
1. **การขับเคลื่อน (Motor Control):** 
   - รูปแบบ: `M,<speedL>,<speedR>\n` (ต้องลงท้ายด้วยการขึ้นบรรทัดใหม่ `\n` เพื่อให้รู้จุดจบคำสั่ง)
   - ตัวอย่าง: `M,50,-50\n` (ล้อซ้ายหมุนไปหน้า 50, ล้อขวาหมุนกลับหลัง -50 ใช้สำหรับการหมุนตัวหรือเลี้ยวแบบรถถัง)
2. **การควบคุมคันเร่ง (Engine Throttle):**
   - รูปแบบ: `T,<percent>\n`
   - ตัวอย่าง: `T,80\n` (กดคันเร่งไปที่ 80%)
3. **การสั่งการเครื่องยนต์ (Engine Status):**
   - รูปแบบ: `E,<status>\n`
   - ตัวอย่าง: `E,1\n` (สั่งเรียกใชังานฟังก์ชัน `startEngine()`), `E,2\n` (เรียกใช้ `powerOn()`), `E,0\n` (เรียกใช้ `powerOff()`)

---

### 3. แนวทางการแยก Ultrasonic ไปใช้ Arduino
ตามไอเดียที่จะใช้ Arduino ตัวเล็กๆ รับภาระไว้อ่านค่าเซ็นเซอร์ Ultrasonic 3 ตัว และส่งค่าให้ ROS2 ตรงๆ เพื่อลดภาระการหน่วงเวลา (Block) ของจังหวะสัญญาณเสียงในบอร์ดหลัก STM32 

**รูปแบบ Protocol จาก Arduino -> ROS2 (คงรูปแบบเดิมจาก STM32 ไว้ จะได้ไม่ต้องแก้โค้ดฝั่งรับของ ROS มาก):**
- รูปแบบ: `U,<dL>,<dC>,<dR>\n`
- ตัวอย่าง: `U,20.5,150.2,40.0\n` (ระยะที่สะท้อนกลับมาหน่วยเป็นเซนติเมตร)

**แนวทางการเขียนโค้ด Arduino:**
- ให้ใช้วิธี **Non-blocking (ไม่ใช้ `delay()` ปกติ ยกเว้นตอน Trigger สั้นๆ ระดับ Microseconds)** ในการอ่านค่าเซ็นเซอร์
- เว้นระยะเวลาพักให้เสียงสะท้อน (Echo ชุดเดิม) หายไปก่อนที่จะสั่งอ่านเซ็นเซอร์ตัวถัดไปประมาณ **35ms** ป้องกัน Crosstalk (เสียงของตัวซ้ายไปเข้าตัวกลาง/ตัวขวา)
- แล้วก็ส่งค่าเซ็นเซอร์ล่าสุด (หรือใส่ค่า -2.0 ไว้ป้องกันกรณีเกินระยะวัด/timeout) ทุกๆ 100ms ออกมาทาง Serial 

---

### 4. แนวทางการใช้งานร่วมกับ ROS2 (Teleop & Serial Node)
ในฝั่งของ ROS2 เพื่อให้ทำงานร่วมกับระบบดังกล่าว เราต้องแบ่งการจัดการออกเป็นฝั่งรับ/ส่ง:

1. **ส่งคำสั่งเข้า STM32 (Teleop Converter Node):**
   - โหนดนี้จะทำหน้าที่ `Subscribe` Topic ที่ชื่อว่า `/cmd_vel` (ใช้ Message แบบ `geometry_msgs/msg/Twist`)
   - นำค่า `linear.x` (ความเร็วเดินหน้า/ถอยหลัง) และ `angular.z` (ความเร็วหมุนซ้าย/ขวา) มาคำนวณผ่านสมการ Kinematics สำหรับหุ่นยนต์ล้อสายพาน (Differential Drive)
   - พอคำนวณได้ความเร็วของล้อซ้ายและล้อขวาแล้ว ให้จับแปลงหรือ Map ค่านั้นให้อยู่ในช่วง `-100` ถึง `100` ตามที่มอเตอร์ของท่านร้องขอ
   - จากนั้นประกอบสตริงรูปแบบ `M,<ค่าล้อซ้าย>,<ค่าล้อขวา>\n` แล้วส่งผ่าน Serial (ไปยังพอร์ตของ STM32) เข้าไป

2. **รับข้อมูลกลับจาก Arduino / STM32 (Serial Reader Node):**
   - ฝั่ง Arduino: โหนดจะคอยอ่านบรรทัด `Serial.readline( )` จากพอร์ต `/dev/ttyUSB` (พอร์ตของ Arduino) 
   - เมื่อเจอบรรทัดที่ขึ้นต้นด้วยตัวอักษร `U,` จะทำการตัดคำ (Split string) เพื่อแยกระยะกะจัดทั้ง 3 ตัวออกมา ส่งเข้าไปในหัวข้อ Topic (เช่น Publish ให้ประเภทข้อมูล `sensor_msgs/msg/Range`) เพี่อส่งให้ส่วนประมวลผล Navigation เอาไปคำนวณการเลี่ยงชน (Obstacle avoidance)
   - สำหรับฝั่ง STM32 (ข้อมูล IMU หรือ Encoder) ก็สามารถทำวิธีเดียวกันแยกตามตัวอักษร `I,` หรือ `D,` แล้วใส่ ROS2 Topics ได้เลย
